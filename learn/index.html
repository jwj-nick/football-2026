🛠️ 이 앱은 어떻게 만들어졌나

옆에 있는 월드컵 앱, 처음엔 어려워 보이지만 사실 네 덩어리로 되어 있어. 이 페이지는 그 덩어리들을 하나씩 뜯어보고, 네가 직접 바꿔볼 미션까지 준다. 목표는 "코드를 외우는 것"이 아니라 "내가 바꾸면 화면이 바뀐다"는 감을 잡는 것. 그게 바이브 코딩의 출발점이야.

1. 큰 그림 — 앱은 네 덩어리

웹 앱은 보통 이 세 가지 역할로 나뉘어:

HTML = 뼈대+ CSS = 꾸미기+ JavaScript = 움직임

이 앱의 파일을 그 역할에 맞춰 나눠 놨어. 한 파일이 한 가지 일만 하도록:

index.html뼈대. 표·버튼·탭이 "어디에 있는지"만 적어둔 곳. 맨 아래에서 아래 4개 파일을 순서대로 불러온다.
css/style.css꾸미기. 색·간격·둥근 모서리. 맨 위 :root 변수만 바꿔도 분위기가 확 바뀐다.
js/data.js재료. 72개 조별리그 + 16개 32강 경기 데이터가 통째로 들어있다.
js/standings.js요리법. 점수를 받아 순위를 계산. 화면은 전혀 안 건드린다.
js/app.js접시에 담기. 데이터+계산 결과를 표로 그리고, 클릭·입력에 반응한다.
💡 왜 굳이 파일을 나눌까? 한 파일에 다 넣으면 처음엔 편하지만, 커지면 어디가 어디인지 못 찾는다. "데이터는 데이터끼리, 계산은 계산끼리" 나눠두면 고칠 곳을 바로 찾고, 한 곳을 고쳐도 딴 데가 안 망가진다. 이걸 "관심사 분리"라고 불러 — 프로 개발자들이 가장 먼저 배우는 습관이야.

2. 데이터: 경기는 어떻게 들어있나

js/data.js를 열면 경기 하나가 이렇게 생겼어:

{
  g:    "A",                   // 조
  iso:  "2026-06-12T11:00",    // 날짜+시간 (정렬·필터의 기준)
  date: "06.12",  dow: "금",   // 보기 좋게 쪼갠 값
  time: "11:00",
  home: "대한민국", away: "체코", // 맞붙는 두 팀
  venue:"과달라하라",            // 경기장
  note: "",                    // 특별 표시 (개막전 등)
  kr:   true                   // 한국 경기면 true → 빨갛게 강조
}

이런 객체(중괄호 묶음)가 배열([ ]) 안에 72개 줄지어 있는 게 전부야. 앱은 이 배열을 처음부터 끝까지 훑으면서 표의 한 줄 한 줄을 만들어 낸다. 경기를 추가하고 싶으면? 이 배열에 객체를 하나 더 넣으면 끝 (→ 미션 2).

3. 계산: 순위는 어떻게 매겨지나

축구 순위 규칙은 간단해. js/standings.js가 정확히 이걸 코드로 옮긴 거야:

  • 이기면 승점 3점, 비기면 1점, 지면 0점
  • 줄 세우는 순서: 승점 → 골득실(득점−실점) → 다득점 → 이름순

코드의 핵심은 딱 이 부분이야:

if (hs > as) {        // 홈팀이 더 넣었으면 → 홈팀 승
  h.W++; a.L++; h.Pts += 3;
} else if (as > hs) { // 원정팀 승
  a.W++; h.L++; a.Pts += 3;
} else {              // 무승부
  h.D++; a.D++; h.Pts += 1; a.Pts += 1;
}

hs는 홈 점수, as는 원정 점수. h.Pts += 3은 "홈팀 승점에 3을 더해라"는 뜻. 네가 결과를 입력하면 이 계산이 다시 돌고, 순위표가 즉시 새로 그려진다. "입력 → 계산 → 화면" 이 흐름이 거의 모든 앱의 심장이야.

4. 저장: 새로고침해도 점수가 남는 이유

점수를 입력하고 브라우저를 껐다 켜도 그대로 남아있지? 비밀은 localStorage야. 브라우저가 가진 작은 메모장이라고 보면 돼. js/app.js에서: 

// 저장: 객체를 글자로 바꿔서 메모장에 적기
localStorage.setItem("wc2026_results", JSON.stringify(results));

// 읽기: 글자를 다시 객체로 되돌리기
results = JSON.parse(localStorage.getItem("wc2026_results"));

컴퓨터는 "객체"를 그대로 저장 못 해서, JSON.stringify로 글자로 바꿔 저장하고, 꺼낼 땐 JSON.parse로 다시 객체로 되돌린다. (이 변환은 앱끼리 데이터를 주고받을 때도 똑같이 쓰여.)

5. 바이브 코딩이란? (Claude와 일하는 법)

바이브 코딩은 "전부 외워서 손으로 짜는 것"이 아니라, AI에게 원하는 걸 말로 설명하고 같이 만들어가는 방식이야. 단, 잘하려면 요령이 있어:

좋은 요청의 3요소

  1. 무엇을 바꾸고 싶은지 (예: "한국 경기를 맨 위에 고정")
  2. 어디인지 힌트 (예: "정렬은 app.js의 renderGS 안")
  3. 어떻게 확인할지 (예: "조별리그 탭에서 한국 3경기가 위에 오면 성공")
나쁜 예 ❌ "앱 더 좋게 만들어줘" → 막연해서 엉뚱한 걸 건드림

좋은 예 ✅ "조별리그 표에서 한국 경기를 항상 맨 위에 보이게 해줘. 정렬 코드는 js/app.js의 renderGS 안에 있어. 바꾼 뒤 조별리그 탭에서 한국 3경기가 위에 오는지 확인해줘."
💡 핵심 습관: 한 번에 하나씩 바꾸고 → 브라우저 새로고침으로 확인 → 잘 되면 다음. 안 되면 "뭐가 안 됐는지" 그대로 말하면 돼. 작게 자주 확인하는 게 제일 빠르다.

6. 미션 — 직접 바꿔보기

아래로 갈수록 어려워져. 각 미션은 "직접 해보기"와 "Claude에게 부탁한다면" 두 길을 다 준다.

LEVEL 1 · 아주 쉬움

미션 1. 앱 색깔 바꾸기

css/style.css 맨 위 :root에서 --primary 값을 #1a3d7c 대신 #7c1a3d(자주색)로 바꾸고 저장 → 브라우저 새로고침. 헤더·버튼·표 머리 색이 한꺼번에 바뀐다.

Claude에게: "이 앱의 기본 색을 자주색 계열로 바꿔줘. css/style.css의 :root 변수만 고치면 돼."
LEVEL 2 · 보통

미션 2. 새 경기 추가하기

js/data.js의 GS 배열에 객체를 하나 더 넣어봐 (예: 친선경기). g, iso, date, dow, time, home, away, venue를 채우면 표·필터·순위에 자동으로 반영된다. 쉼표(,) 빠뜨리지 않기!

Claude에게: "js/data.js에 6월 12일 13:00 '대한민국 vs 일본' 친선경기를 A조 경기로 추가해줘."
LEVEL 2 · 보통

미션 3. 한국 경기를 맨 위로

js/app.js의 renderGS 정렬 부분을 손봐서 한국 경기(d.kr === true)가 항상 위에 오게 만들어봐. "이긴 팀 초록 강조"처럼 작은 규칙을 더 추가해도 좋아.

Claude에게: "조별리그 표에서 한국 경기를 항상 맨 위에 고정해줘. 정렬은 app.js의 renderGS 안에 있어."
LEVEL 3 · 도전

미션 4. "내 응원팀" 기능 만들기

버튼을 눌러 좋아하는 팀을 고르면 그 팀 경기를 강조(별표/색)해서 보여주는 기능. HTML(버튼) + CSS(강조색) + JS(상태 저장)를 다 건드려야 해서, 지금까지 배운 걸 다 쓰는 종합 미션이야.

Claude에게: "응원팀을 하나 고르면 그 팀 경기에 ⭐를 붙이고, 선택을 localStorage에 저장하는 기능을 추가해줘. 한국 강조(krrow) 방식과 비슷하게."
LEVEL 3 · 도전 (큰 확장)

미션 5. 실시간 경기 결과 자동으로 가져오기

지금은 점수를 손으로 넣지? 이걸 인터넷에서 자동으로 가져오게 만드는 게 다음 큰 산이야. 이건 그냥 파일 몇 줄로 안 되고, "서버"라는 새 개념이 필요해. 왜냐하면:

  • 축구 데이터 API는 비밀 열쇠(API 키)가 있어야 쓸 수 있는데, 그걸 브라우저 코드에 넣으면 누구나 훔쳐본다.
  • 그래서 열쇠는 서버에 숨기고, 브라우저는 우리 서버에게만 물어보게 만든다.

전체 설계와 단계별 가이드는 프로젝트 폴더의 WORKFLOW.md(API 상세)와 CLAUDE_CODE_MIGRATION.md에 이미 정리돼 있어. 준비되면 그걸 보고 도전!

Claude에게: "WORKFLOW.md의 'API 연동 가이드'를 보고, football-data.org에서 경기 결과를 가져오는 가장 단순한 버전을 같이 만들어줘. 단계별로, 하나씩 확인하면서."

7. 다음 레벨

이 앱은 일부러 빌드 도구 없이(npm·React 없이) 만들었어. 파일을 더블클릭하면 바로 켜지니까 "바꾸면 바로 보이는" 재미를 빠르게 느끼라고. 충분히 익숙해지면 다음 단계가 기다린다:

  • GitHub Pages 배포 — 내 앱을 인터넷 주소로 만들어 친구·가족에게 공유 (→ README.md)
  • React + Vite로 옮기기 — 앱이 커져도 깔끔하게 관리하는 "정식" 방식 (→ CLAUDE_CODE_MIGRATION.md에 전체 계획 있음)
  • 실시간 데이터·캘린더 내보내기 등 기능 확장
🎯 가장 중요한 건 "작게 바꾸고, 확인하고, 또 바꾸기"를 반복하는 것. 완벽하게 이해하고 시작하는 사람은 없어. 일단 미션 1부터 손으로 바꿔봐!
