📊 데이터로 즐기는 축구

요즘 축구는 '감'만으로 보지 않아. 슈팅 몇 개, 점유율 몇 %, 그리고 xG(기대득점)까지 — 숫자로 보면 같은 경기가 다르게 보인다. 그리고 사실, 이 앱이 결과를 받아 순위를 계산하는 것도 아주 작은 데이터 분석이야. 그 이야기를 해보자.

"한 골 차로 졌지만 사실 더 잘한 건 우리였다" — 이런 말, 들어봤지? 예전엔 그냥 느낌이었지만, 지금은 숫자로 증명할 수 있어. 그 도구가 바로 데이터다. 이 페이지를 다 보면 직접 만들기와 데이터 분석가 되기로 자연스럽게 이어진다.

1. 기본 스탯부터 — 경기가 끝나면 뭐가 남나

중계 화면 구석이나 경기 요약에서 자주 보이는 숫자들이야. 하나씩 한 줄로 정리했어.

스탯뜻뭘 말해주나
슈팅(Shots)골을 노리고 찬 횟수공격을 얼마나 시도했나
유효슈팅 (On Target)?골문 안쪽으로 간 슛위협적인 슛이 몇 개였나
점유율 (Possession)?공을 가진 시간 비율(%)경기를 누가 더 끌고 갔나 (단, 점유율 ≠ 승리)
패스 성공률찬 패스 중 동료에게 간 비율(%)볼을 얼마나 안정적으로 다뤘나
코너킥얻은 코너의 수상대 진영에서 얼마나 압박했나(대략)
파울 / 카드반칙 횟수, 경고·퇴장경기가 얼마나 거칠었나, 규율
💡 함정 주의: 점유율이 높다고 항상 이기는 게 아니야. 공을 적게 가져도 결정적인 순간에 효율적이면 이긴다. 그래서 "얼마나 했나(양)"뿐 아니라 "얼마나 좋은 기회였나(질)"를 보는 스탯이 필요해. 그게 다음 주인공, xG다.

2. xG(기대득점)란? — 운이었나, 실력이었나

이 페이지에서 제일 중요한 개념. 천천히 가자.

💬 xG(expected goals, 기대득점)란? 한 경기에서 나온 모든 슛마다 "이게 골이 될 확률"을 매겨서, 그 확률을 전부 더한 값이야. 골대 바로 앞 텅 빈 골문이면 0.9(거의 골), 하프라인 근처 무리한 슛이면 0.02(거의 안 들어감) 식으로. 그 확률들을 다 합치면 "이 팀은 만든 기회로 보면 평균 몇 골 정도를 넣었어야 한다"가 나온다.

예를 들어 한 슛의 골 확률을 이렇게 매겼다고 하자(숫자는 이해용 예시야):

슛 상황골이 될 확률(xG)
페널티킥약 0.76
골문 바로 앞 빈 찬스약 0.6 ~ 0.9
박스 안 보통 슛약 0.1 ~ 0.2
먼 거리 / 각 안 좋은 슛약 0.02 ~ 0.05
📌 핵심 활용: 한 경기에서 우리 팀 xG가 2.3인데 실제로는 1골만 넣었다면? "좋은 기회를 충분히 만들었는데 마무리에서 조금 안 풀렸다"는 신호일 수 있어 — 그러면 다음 경기엔 더 들어갈 가능성도 있는 거지. 반대로 xG 0.4인데 3골을 넣었다면 "오늘은 운(또는 슈퍼 골)이 좋았다"에 가깝고, 그게 매번 반복되긴 어렵다고 본다.
⚠️ 과장 금지: xG는 점쟁이가 아니야. 슛의 위치·각도·상황을 바탕으로 한 확률 추정일 뿐, "이 팀이 다음에 무조건 이긴다"를 말해주지 않아. 모델·데이터 회사마다 계산법이 조금씩 달라서 같은 경기 xG가 약간 다르게 나오기도 해. 그래서 xG는 '하나의 참고 지표'로 보는 게 맞아. (이런 "단정하지 않기"가 데이터를 다루는 사람의 기본 태도다.)

3. 이 데이터는 어떻게 모으나

이 많은 숫자는 저절로 생기지 않아. 경기장에서 누군가(또는 무언가)가 측정하고 기록해.

🎥

다중 카메라

경기장 곳곳에 여러 대의 카메라가 모든 각도에서 경기를 찍는다.

🛰️

선수·공 트래킹

영상·센서로 선수 22명과 공의 위치를 초당 수십 번 기록한다(좌표 데이터).

⌨️

데이터 입력 요원

사람이 실시간으로 "패스·슛·태클" 같은 사건에 태그를 붙여 기록한다.

이렇게 모인 위치 좌표 + 사건 기록을 컴퓨터가 분석해서 점유율·패스성공률·xG 같은 숫자로 바꾼다. 바로 이걸 직업으로 하는 사람이 있어 — 스포츠 데이터 분석가.

📌 아빠 분야와 연결: 카메라·센서로 raw 신호를 모으고 → 의미 있는 숫자로 가공하고 → 판단에 쓰는 흐름은, 아빠가 일하는 데이터·AI·반도체 분야와 똑같은 뼈대야. 축구든 반도체든, "데이터에서 인사이트를 뽑는" 일의 원리는 닮아 있다. → 스포츠 데이터 분석가 / 테크 커리어 보러 가기

4. 우리 앱도 데이터다 🎯

거창한 회사 이야기 같지? 그런데 이 앱도 정확히 같은 일을 작게 하고 있어. 네가 점수를 입력하면, standings.js가 그걸 받아 순위표로 바꿔준다.

핵심은 딱 두 가지 규칙이야 (실제 우리 코드의 일부):

if (hs > as) {        // 홈팀이 더 많이 넣으면 → 홈팀 승
  h.W++; a.L++; h.Pts += 3;     // 이긴 팀 승점 +3
} else if (as > hs) { // 원정팀 승
  a.W++; h.L++; a.Pts += 3;
} else {              // 같으면 무승부
  h.D++; a.D++; h.Pts += 1; a.Pts += 1;   // 둘 다 +1
}

그리고 순위를 매길 땐 이 순서로 정렬해(이것도 실제 코드):

// 승점 높은 순 → 골득실 높은 순 → 다득점 순 → 이름순
arr.sort(function (x, y) {
  return (y.Pts - x.Pts)
      || (y.GD - x.GD)
      || (y.GF - x.GF)
      || x.t.localeCompare(y.t, "ko");
});
💬 골득실(GD, Goal Difference)이란? 득점에서 실점을 뺀 값이야(예: 5골 넣고 2골 먹으면 +3). 승점이 같을 때 누가 위인지 가르는 첫 번째 기준이지. 우리 코드에선 GD = GF - GA로 계산한다.
📌 이게 바로 "데이터 → 정보" 변환이다. 네가 입력한 건 그냥 숫자 쌍(2:1, 0:0…)인데 = 데이터. 규칙(승점·골득실)을 적용해 정렬하면 "지금 1위는 누구"라는 정보가 된다. 프로 구단이 xG로 하는 일과 원리는 똑같아 — 규모만 다를 뿐이야. → 이 계산이 어떻게 만들어졌는지 '배우기'에서 보기

5. 눈으로 보기 (시각화)

표의 숫자도 좋지만, 그림으로 보면 "어느 팀이 어디서 강한지"가 한눈에 들어와. 가상의 두 팀을 다섯 가지 스탯으로 비교한 레이더 차트야 (인터넷 연결 시 그림으로 표시).

가상 두 팀 스탯 비교 (예시 데이터)

레이더 차트는 인터넷 연결 시 표시됩니다. 안 보이면 아래 표를 보세요. (값은 0~100으로 환산한 예시)

지금까지 이야기를 한 줄로 그리면 이렇게 흐른다 (인터넷 연결 시 그림으로 표시):

6. 미션 — 직접 해보기 🚀

읽기만 하면 안 남아. 둘 중 하나(또는 둘 다!)를 골라 Claude에게 부탁해보자.

쉬움

① 간단 스탯 한 개 더 보여주기

지금 순위표엔 승/무/패·승점·골득실이 있어. 여기에 "평균 득점"(= 총 득점 ÷ 경기 수) 같은 간단한 스탯을 한 칸 더 넣어보자. 우리가 4단원에서 본 것처럼, standings.js는 이미 GF(득점)와 P(경기 수)를 알고 있으니 나누기 한 번이면 돼.

🗣️ Claude에게 이렇게 부탁해봐:
"standings.js에서 각 팀의 '평균 득점(총 득점 ÷ 경기 수)'을
 계산해서 순위표에 한 칸 더 보여주고 싶어. 경기를 0번 한 팀은
 0으로 나누지 않게 조심해서, 초보가 읽게 한글 주석도 달아줘."
보통

② 두 팀 스탯으로 레이더 차트 그리기

5단원의 레이더 차트를 네 손으로 만들어보자. 좋아하는 두 팀(또는 가상의 팀)의 스탯 다섯 개(슈팅·점유율·패스·수비·xG)를 0~100으로 정하고, Chart.js radar로 그려. 핵심은 이 페이지처럼 오프라인 폴백(표)도 꼭 같이 두는 거야.

🗣️ Claude에게 이렇게 부탁해봐:
"Chart.js radar 차트로 두 팀의 스탯 5개(슈팅·점유율·패스·수비·xG)를
 비교하는 작은 HTML을 만들어줘. 우리 explore 페이지처럼 인터넷이
 없을 때 보이는 폴백 표도 같이 넣고, 한글 주석으로 설명해줘."
💡 미션을 끝내면 스스로에게 물어봐: "이 숫자가 정말 그 뜻이 맞을까? 예외는 없을까?" (예: 경기 0번 한 팀의 평균 득점은?) 이렇게 의심하고 확인하는 습관이 진짜 데이터 분석가의 핵심이야.
